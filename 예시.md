# 실전 예시: 같은 부품으로 다른 몬스터 만들기 (주황버섯 vs 뿔버섯)

> [코드 규범](./코드%20규범.md)의 "상속 대신 조합" 원칙을 실제로 보여주는 예시입니다.
> 주황버섯1(돌진 패턴, HP 100·이속 3·힘 3)과, 그와는 다른 뿔버섯2를 만들어 봅니다.
> 코드 예시와 API는 메이플 월드 위키(공격과 피격 / 행동 트리 / MovementComponent / 상태 제어)를 근거로 작성하였습니다.

---

## 결론부터

주황버섯과 뿔버섯은 **서로 다른 클래스가 아닙니다.** **같은 부품(컴포넌트·행동 노드)을 다르게 조합하고, 데이터 값만 바꾼 것**입니다. 이게 우리가 규범에서 말한 "상속 대신 조합"이 실제로 작동하는 모습입니다.

---

## ❌ 정통 OOP 본능 (메이플에선 함정입니다)

머릿속에 이런 그림이 떠오르면 위험 신호입니다.

```
class Monster { HP, Speed, Str; Move(); TakeDamage(); }
class 주황버섯 : Monster { override 패턴() = 돌진 }
class 뿔버섯   : Monster { override 패턴() = 점프공격 }
```

상속 트리로 가면 패턴이 늘 때마다 클래스가 폭발하고, 메이플 엔진(엔티티-컴포넌트 구조)과 맞지 않아 엔진과 싸우게 됩니다. 메이플은 "클래스 계층"이 아니라 "엔티티에 컴포넌트를 붙이는 조합"으로 돌아가기 때문입니다.

---

## ✅ 메이플 방식: 부품(모듈)을 만들고 → 조립 + 데이터만 다르게

위키에서 확인한 실제 부품들입니다.

- **`HitComponent`** (네이티브) → Extend해서 피격 처리 (`OnHit`)
- **`MovementComponent`** (네이티브) → 이동/방향 제어 (`MoveToDirection`, `Stop`, `IsFaceLeft`)
- **`AIComponent` + `BTNodeType`** (네이티브) → 행동 트리로 패턴(돌진 등) 구현
- 여기에 규범대로 **데이터 모듈**(`MonsterStatComponent`)을 하나 더 만듭니다.

---

### 1) 재사용 부품 ① — 스탯 모듈 (코드/데이터 분리)

HP·이속·힘은 **코드가 아니라 데이터**입니다. 같은 컴포넌트를, 값만 다르게 붙입니다.

```lua
-- MonsterStatComponent  (모든 몬스터가 공유하는 단 하나의 스탯 모듈)

Property:
    [None]
    number MaxHP = 100      -- 프로퍼티 에디터에서 몬스터별로 편집합니다
    [None]
    number MoveSpeed = 3
    [None]
    number Str = 3

    [Sync]
    number CurrentHP = 100  -- 서버 권위 상태입니다

Method:
    [server only]
    void OnInitialize()
    {
        self.CurrentHP = self.MaxHP
    }
```

> 주황버섯도 뿔버섯도 **이 스크립트를 똑같이** 붙입니다. 코드를 복사하지 않습니다.

### 2) 재사용 부품 ② — 피격 모듈 (HitComponent 확장)

```lua
-- MonsterHit  (HitComponent를 Extend, 모든 몬스터 공용)

Method:
    override void OnHit(Entity attacker, int damage, boolean isCritical, string attackInfo, int hitCount)
    {
        __base:OnHit(attacker, damage, isCritical, attackInfo, hitCount)

        local stat = self.Entity.MonsterStatComponent
        stat.CurrentHP = math.max(0, stat.CurrentHP - damage)

        if stat.CurrentHP <= 0 then
            self.Entity:SendEvent(MonsterDiedEvent())   -- 죽음은 이벤트로만 알립니다
        end
    }
```

> 이 모듈은 드랍·이펙트·점수 처리를 **모릅니다.** "죽었다"는 사실만 이벤트로 알립니다. 드랍을 추가하고 싶으면 `MonsterDiedEvent`를 구독하는 별도 모듈/커넥터를 붙이면 됩니다.

### 3) 재사용 부품 ③ — 돌진 패턴을 "행동 노드"로 (BTNodeType)

돌진을 주황버섯 스크립트 안에 박지 않고 **독립된 행동 노드**로 만듭니다. 나중에 다른 몬스터도 돌진을 쓰고 싶으면 이 노드만 꽂으면 됩니다.

```lua
-- ChargeBTNode  (BTNodeType, "돌진" 패턴 하나만 담당)

Property:
    [None]
    number ChargeSpeed = 2     -- 데이터: 돌진 속도
    [None]
    number ChargeTime = 1      -- 데이터: 돌진 지속 시간
    number _elapsed = 0

Method:
    void OnInit()
    {
        self._elapsed = 0
    }

    any OnBehave(number delta)
    {
        local move = self.ParentAI.Entity.MovementComponent   -- 노드는 ParentAI로 엔티티에 접근합니다
        local dir  = move:IsFaceLeft() and Vector2(-1, 0) or Vector2(1, 0)

        move:MoveToDirection(dir * self.ChargeSpeed, 0)        -- 방향 이동 (위키 정석)

        self._elapsed += delta
        if self._elapsed < self.ChargeTime then
            return BehaviourTreeStatus.Running   -- 아직 돌진 중입니다
        end

        move:Stop()                              -- 돌진이 끝나면 정지합니다
        return BehaviourTreeStatus.Success
    }
```

### 4) 조립 — AIComponent에서 "어떤 패턴을 쓸지"만 결정

`AIComponent`를 Extend한 곳에서 트리를 짭니다. 핵심은 **주황버섯과 뿔버섯의 코드 차이가 사실상 한 줄**이라는 점입니다.

```lua
-- MonsterAIComponent (AIComponent Extend, 트리 조립 = 접착제 역할)

Property:
    [None]
    string PatternNodeName = "ChargeBTNode"   -- 데이터: 어떤 패턴 노드를 쓸지

Method:
    [server only]
    void OnBeginPlay()
    {
        self:CreateBehaviourTree()
    }

    [server only]
    void CreateBehaviourTree()
    {
        local root  = SelectorNode("Root")
        local chase = self:CreateNode("ChaseBTNode", "chase")   -- 공용: 플레이어 추적

        -- ▼▼ 몬스터를 가르는 단 하나의 지점입니다 ▼▼
        local pattern = self:CreateNode(self.PatternNodeName, "pattern")
        -- ▲▲ 주황버섯: "ChargeBTNode" / 뿔버섯: "JumpAttackBTNode" ▲▲

        local seq = SequenceNode("battle")
        seq:AttachChild(chase)
        seq:AttachChild(pattern)

        root:AttachChild(seq)
        self:SetRootNode(root)
    }
```

`PatternNodeName`을 `[None]` 데이터 프로퍼티로 빼두었기 때문에, **코드는 손도 대지 않고** 프로퍼티 에디터에서 주황버섯 모델엔 `"ChargeBTNode"`, 뿔버섯 모델엔 `"JumpAttackBTNode"`만 적으면 끝납니다.

---

## 그래서 주황버섯1 vs 뿔버섯2 — 차이는 **데이터 + 조합**뿐입니다

둘 다 새 스크립트를 **한 줄도 만들지 않습니다.** 모델(Model)에 같은 부품을 붙이고 프로퍼티 에디터에서 값만 다르게 줍니다.

| | **주황버섯1** (Model) | **뿔버섯2** (Model) | 어디서 정하나 |
|---|---|---|---|
| `MonsterStatComponent` | MaxHP=100, MoveSpeed=3, Str=3 | MaxHP=150, MoveSpeed=2, Str=5 *(예시)* | 프로퍼티 에디터 (데이터) |
| `MonsterHit` | ✅ (공용, 동일) | ✅ (공용, 동일) | 공용 모듈 |
| `MovementComponent` | ✅ | ✅ | 네이티브 |
| `MonsterAIComponent`의 `PatternNodeName` | `"ChargeBTNode"` (돌진) | `"JumpAttackBTNode"` (점프공격) | 프로퍼티 에디터 (데이터) |
| 새로 작성한 코드 | **0줄** | **0줄** | — |

→ 주황버섯에 돌진이 있는 이유는 "돌진 클래스를 상속해서"가 아니라 **AI 트리에 `ChargeBTNode`를 꽂았기 때문**이고, 뿔버섯은 그 자리에 다른 노드를 꽂았을 뿐입니다.

---

## 이 예시가 규범과 어떻게 맞물리나

- **재사용 모듈 70%** → `MonsterStatComponent`, `MonsterHit`, `ChaseBTNode`, `ChargeBTNode` (다른 몬스터에도 그대로 씁니다)
- **접착제 30%** → `MonsterAIComponent` (어떤 노드를 꽂을지 조립만 합니다)
- **코드와 데이터 분리** → HP·이속·힘·돌진 속도·패턴 종류가 전부 `[None]` 데이터입니다
- **이벤트 연결** → 죽음은 `MonsterDiedEvent`로만 알립니다 (스탯/피격 모듈이 드랍·이펙트를 모릅니다)
- **상속 대신 조합** → 몬스터 종류 = 클래스가 아니라 **부품 조합표**입니다

---

## 검증 기준

새 몬스터(예: 슬라임)를 추가할 때, **코드를 짜지 않고 새 Model에 부품 조합 + 수치 입력만으로** 끝난다면 설계가 제대로 작동하는 것입니다. 만약 새 몬스터마다 새 스크립트를 복사하고 있다면, 그건 부품화가 덜 된 것이니 공통 부분을 모듈로 다시 빼야 합니다.
