# GitWiki 설계 스펙 (콘텐츠 무관 프레임워크)

> 목적: 단일 원본 자료를 **GitHub 위키(GitWiki)** 형태로 변환·배포하기 위한 재사용 가능한 구조 설계.
> 이 문서는 특정 주제와 무관한 **틀(framework)** 만 정의한다. 어떤 주제의 위키든 이 골격에 끼워 쓸 수 있다.
> (이 저장소의 실제 구현 예시는 `wiki/`(분할본) → `gitwiki/`(배포본) 파이프라인.)

---

## 0. 설계 제1원칙

> **본문은 무손실 보존(요약 금지). 매번 로드하는 건 작은 인덱스뿐, 본문은 필요할 때만 연다.**

"통째 보관 ≠ 통째 로드"를 분리한다. 이 한 줄이 아래 모든 구조를 결정한다.

---

## 1. 3단계 표현 계층 (단방향 파이프라인)

```
[원본 1개]  ──split──▶  [분할본 트리]  ──build──▶  [GitWiki 배포본]
 source                  working set                publish target
```

| 계층 | 형태 | 역할 |
|---|---|---|
| **원본(source)** | 단일 거대 파일 | 진실의 원천. 평소엔 안 읽고 `grep` 검색용 |
| **분할본(working)** | 계층형 폴더 + 문서 N개 | 작업·조회·검증 대상. 한 문서 = 한 파일 |
| **배포본(gitwiki)** | 플랫 페이지 + 네비 파일 | GitHub `.wiki` 호스팅용. 분할본에서 스크립트로 파생 |

- **단방향**: 원본만 고치면 `split → build`로 아래가 재생성된다. 배포본을 손으로 고치지 않는다.
- 배포본 생성기는 분할본의 본문을 **그대로 복사**하고 네비/배너만 입힌다.

---

## 2. GitWiki 배포본 레이아웃 (핵심)

GitHub 위키는 **플랫 네임스페이스 + 특수 네비 파일** 규약을 따른다.

```
gitwiki/
  Home.md          # 랜딩 페이지 (위키 첫 화면)
  _Sidebar.md      # 모든 페이지에 고정 노출되는 사이드바
  _Footer.md       # 모든 페이지 공통 꼬리말
  <제목-슬러그>.md  # 페이지들 (폴더 없이 평평하게)
```

규약 핵심:
- `Home` / `_Sidebar` / `_Footer` 는 GitHub 위키가 **자동 인식**하는 예약 파일명.
- 페이지는 **폴더 없이 평평**하게 둔다(하위폴더는 링크가 까다로워짐).
- 분류(카테고리)는 경로가 아니라 **`_Sidebar`와 `Home`이 표현**한다.

---

## 3. 명명 / 슬러그 규칙

- 페이지 파일명 = **제목의 슬러그** + `.md`.
- 슬러그화:
  1. 공백 → `-`
  2. GitHub 위키 금지문자 제거: `\ / : * ? " < > | # %`
  3. 연속 `-` 축약, 양끝 `-` 제거
  4. **충돌 시 `-2`, `-3` 접미사**로 유일성 보장
- (분할본 쪽은 정렬 안정성을 위해 `NN-` 숫자 접두사 + 폴더 계층을 쓰지만, **배포본 페이지명에는 숫자 접두사를 넣지 않는다** — 제목 = 페이지명이 되도록.)

---

## 4. 하이퍼링크 / 네비게이션 설계

링크는 **한 방향으로 흐른다: 지도 → 페이지** (페이지끼리는 안 엮음 = 트리, 그래프 아님).

### 4.1 Home.md (랜딩 = 색인)
2단 구성:
```markdown
# <위키 제목>
<한 문단 소개> + <"더 알고 싶으면 어디서 검색" 안내(선택)>

## 분류
- [<분류명>](#분류명-앵커)     ← 인페이지 점프(목차)
...

## <분류명>
- [<페이지 제목>](<슬러그>)     ← 실제 페이지 링크
...
```
- 상단 분류 목록은 **`#앵커` 인페이지 점프**.
- 하단은 분류별 **전체 페이지 목록**.

### 4.2 _Sidebar.md (영구 네비)
```markdown
### <위키 제목>
[🏠 Home](Home)

**<분류명>**
- [<페이지 제목>](<슬러그>)
...
```
- 모든 페이지 옆에 항상 떠 있음.

### 4.3 링크 표기
- **GitWiki 페이지 링크 = `[표시제목](슬러그)`** — 폴더·확장자 없음.
- 인페이지 점프 = `[표시](#앵커)`.
- (참고: GitHub은 `[[제목]]` 위키링크도 지원하지만, 제목에 금지문자가 있으면 슬러그와 안 맞을 수 있으므로 **명시적 `[제목](슬러그)` 링크를 권장**.)

---

## 5. 목록(리스트) 표현 패턴

모든 목록은 한 줄 동일 포맷:
```
- [<제목>](<링크>) — <한 줄 설명>
```
- **한 줄 설명은 본문에서 자동 추출**(각 문서 첫 실제 문단을 잘라 N자 컷).
- 분류는 `##` 헤더로 그룹핑, 그 아래 평평한 목록.
- **2단 깊이만** 보여준다(트리를 끝까지 펼치지 않음 → 인지 부하 관리).

---

## 6. 페이지 1개의 내부 골격

```markdown
# <제목>

> <배너 블록: 출처/검색 안내 등 메타 영역>
> _<분류 · 출처>_

---

<본문 — 원본 그대로, 가공 0>
```
- 상단 **배너 블록 + `---` 구분선**으로 "메타 영역"과 "본문 영역"을 시각적으로 분리.
- 배너에는 "더 자세한 원문은 어디서/무엇으로 검색"하라는 안내를 넣을 수 있다(페이지 제목이 외부 원문 제목과 일치할 때 특히 유용).

---

## 7. 분할(파싱) 설계 — 원본을 어떻게 자르나

콘텐츠 무관 일반화:
1. **신뢰할 구분자(delimiter) 1개**를 찾아 레코드(문서) 경계를 정한다.
2. 경계의 **줄번호만** 뽑아 슬라이싱한다(본문을 메모리/컨텍스트에 올리지 않음 → 거대 원본도 한 번에 처리).
3. **제목 리스트는 별도의 권위 소스**로 확보한다(구분자가 제목을 안전하게 주지 못할 때).
4. 본문은 **줄 범위 그대로 복사**(요약·재작성 0).
5. 헤더는 메타데이터로 승격(`#`), 마커·중복 네비 줄은 버린다.
6. **무손실 검증**: 원본 본문 줄 멀티셋 = 분할본 줄 멀티셋(버려진 건 마커/제목줄/중복네비뿐임을 확인).

---

## 8. 조회(런타임) 설계 — AI/사람이 쓸 때

1. **인덱스(작은 지도)부터** 로드.
2. 키워드로 후보 1~3개 선택.
3. 애매하면 **원본을 `grep`**으로 위치 보강.
4. 고른 **그 파일만** 로드해 답하고 **출처(경로)를 밝힌다**.
5. 인덱스/위키에 없으면 "없음"을 명시하고, 필요 시 외부 검색어를 안내.

→ "전체 적재"가 아니라 "검색 후 부분 적재" 모델.

---

## 9. 생성 파이프라인 (도구)

| 단계 | 입력 → 출력 | 책임 |
|---|---|---|
| `split` | 원본 1개 → 분할본 트리 + 인덱스 | 무손실 분할, 제목·분류 부여, 인덱스 생성 |
| `build` | 분할본 트리 → gitwiki 배포본 | 플랫화, 슬러그화, 배너·네비(Home/_Sidebar/_Footer) 생성 |

- 두 도구 모두 **본문을 가공하지 않는다**(요약 금지 원칙 유지).
- 원본이 바뀌면 `split → build` 재실행으로 전체 재생성.

---

## 10. 한 문장 요약

> **단일 원본을 무손실 분할 → "계층 경로(작업본) / 플랫 슬러그(GitWiki)" 두 네임스페이스로 노출하고, 항상 작은 인덱스만 먼저 읽은 뒤 본문은 grep+부분로드로 가져오는, 트리형 단방향 링크 구조.**
