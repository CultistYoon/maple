# 제가 생각하는 객체지향 코드 규범

> 이 문서는 "재사용 가능한 모듈식 설계 / 모듈 간 연결 / 코드·데이터 분리 / 실행 순서 제어 / 시뮬레이션·뷰 분리"라는 나님 허윤의 설계 철학을 예시를 넣어 썼습니다.
>
> 코드 예시는 MSW 스크립트 에디터의 표기(`Property:` / `Method:` / `Event Handler:` 섹션, 타입 우선 선언, 실행 공간 어트리뷰트)를 그대로 사용하였습니다.

---

## 0. 꼭 알아야하는 메이플 월드 에디터 용어(대박 고양이급)

| 용어 | 의미 |
|---|---|
| **Script Component** | 엔티티에 부착되는 로직 단위입니다. `Create Scripts > Create Component`로 만듭니다. 이 문서에서 "모듈"은 보통 이걸 가리킵니다. |
| **Property** | 컴포넌트의 상태/데이터 값입니다. `[Sync]`(서버→클라 동기화) 또는 `[None]`(동기화 안 함)으로 선언하며, 프로퍼티 에디터에 노출됩니다. |
| **EventType** | 크리에이터가 직접 정의하는 커스텀 이벤트입니다. `self.Entity:SendEvent()`로 발송합니다. |
| **Connector Component** | 여러 모듈을 이어 붙이기만 하는 접착제 전용 스크립트입니다. |
| **`_TimerService`** | 지연/반복 실행을 예약하는 서비스입니다. 타이머는 전부 여기로 처리합니다. |
| **`OnInitialize` / `OnBeginPlay` / `OnUpdate` / `OnSyncProperty`** | 시스템이 정해진 시점에 자동 호출하는 기본 이벤트 함수입니다. (4번 참고) |

## 핵심 철학

### 1. 재사용 가능한 모듈식 설계

- 플레이어 이동, 체력 시스템, 적 생성기 등 각 기능을 **독립적인 Script Component**(레고 블록)로 만듭니다.
- 각 컴포넌트는 다른 컴포넌트가 없어도 자체적으로 동작해야 합니다. (꼭 필요하면 최소한의 '연결'만 허용합니다)
- **재사용 검증 기준:** 이 컴포넌트를 *다른 월드/다른 엔티티에 그대로 붙였을 때* 동일하게 동작하는가? 특정 엔티티 이름이나 GUID, 특정 맵 구조에 의존한다면 모듈이 아닙니다.
- 모듈은 **자신과 자신의 Property/데이터만** 알아야 합니다. 외부의 구체적인 컴포넌트를 직접 부르지 않습니다.

### 2. 모듈끼리의 '연결'

- 독립된 블록들은 서로 연결되어야만 합니다. "재사용 모듈 70% + 하드코딩 30%" 비율을 권장합니다.
- **나쁜 연결:**
  - `_EntityService:GetEntity("a22396e1-...")` 처럼 **GUID/경로를 모듈 내부에 하드코딩**해 다른 엔티티를 끌어오는 것입니다.
  - 모듈이 외부의 특정 컴포넌트(`someEntity.UIManagerComponent`)를 직접 호출하고 그 결과에 의존하는 것입니다.
  - 뷰(UI/사운드/이펙트) 호출이 로직 모듈 안에 섞이는 것입니다.
- **좋은 연결:**
  - **Entity Event System** — 모듈은 이벤트만 쏘고(`self.Entity:SendEvent(event)`), 반응은 구독자가 결정합니다.
  - **Connector Component** — 모듈 참조를 Property로 주입받아 이벤트를 중계/조율하는 접착제 전용 스크립트입니다.
  - **`[Sync]` 프로퍼티** — 서버 로직이 값만 바꾸면 클라이언트가 알아서 반응합니다.
- > ⚠️ `self.Entity.XxxComponent`로 **같은 엔티티에 붙은 컴포넌트를 참조하는 것 자체는 정상**입니다. 문제가 되는 건 *하드코딩된 외부 엔티티 조회*와 *모듈 간 높은 의존성*입니다. 판단 기준은 언제나 "이 모듈을 떼어 다른 곳에 붙여도 깨지지 않는가"입니다.

### 3. 코드와 데이터를 분리하라

- **코드** = 게임 로직, **데이터** = 밸런싱 수치·번역 텍스트·레벨 구성입니다.
- MSW에서 데이터를 코드에서 분리하는 수단은 다음과 같습니다.
  - **`[None]` 데이터 프로퍼티** — 프로퍼티 에디터에서 엔티티별로 수치를 편집합니다(코드 수정 불필요).
  - **LocaleDataSet** — 번역/표시 텍스트, 로컬라이징되는 리소스 키를 담습니다.
  - **DataStorage** — 영속 데이터를 담습니다.
- **나쁜 예:** UI 텍스트를 코드에 직접 박는 것입니다 → `textComponent.Text = "E를 눌러 상호작용"`
- **좋은 예:** LocaleDataSet에 키로 정리합니다 → `textComponent.Text = _LocalizationService:GetText("UI_INTERACT_HINT")`

### 4. 실행 순서를 제어하라

- 스크립트 실행 순서는 그 자체로 스파게티의 원인입니다. MSW는 **명확한 라이프사이클**을 제공하므로 이를 정확히 이해하고 씁니다.
- **컴포넌트 간 `OnBeginPlay` 호출 순서는 보장되지 않습니다.** 다른 컴포넌트가 참조할 값은 `OnBeginPlay`가 아니라 **`OnInitialize`에서 할당**합니다.
- 타이머/지연은 **`wait()`나 코루틴 오용으로 흉내 내지 말고 `_TimerService`로** 제어합니다. (아래 컨벤션 ⚙️ 참고)

### 5. 시뮬레이션과 뷰 분리하기

- **시뮬레이션(로직):** 눈에 안 보이는 순수 데이터·로직입니다. MSW에서는 **서버(`[server only]`)**가 권위(authority)를 갖습니다. HP, 좌표, 상태 값 등이 여기 해당합니다.
- **뷰(표현):** 시뮬레이션 값을 *읽어서* 애니메이션·이펙트·사운드·UI로 보여줍니다. MSW에서는 **클라이언트(`[client only]`)**가 담당합니다.
- 연결 고리는 **`[Sync]` 프로퍼티 + `OnSyncProperty`**입니다. 서버가 값을 바꾸면 클라이언트로 동기화되고, 클라이언트는 동기화 시점에 뷰를 갱신합니다.
- 뷰는 시뮬레이션 값을 **읽기만** 합니다. 클라이언트에서 권위 값을 직접 바꾸지 않습니다(동기화 방향은 서버→클라이언트).

## 🎯 네이밍 컨벤션

메이플 월드 스크립트 에디터의 표기와 Lua 관례를 따릅니다.

### Script Component / EventType / 데이터 이름

```
-- 모듈 (재사용 가능한 Script Component)
HealthComponent
MovementComponent
EnemySpawnerComponent

-- 커넥터 (접착제 전용 Script Component)
PlayerHealthConnector
EnemySpawnConnector

-- 시뮬레이션(서버 권위 로직) / 뷰(클라이언트 표현) 를 한 컴포넌트로 나눌 때
CombatComponent        -- 서버 권위 로직
CombatViewComponent    -- 클라이언트 표현

-- 커스텀 이벤트 타입 (EventType)
PlayerDamagedEvent
EnemyDiedEvent
SunriseEvent

-- 데이터 (LocaleDataSet)
UI_LocaleDataSet
Dialogue_LocaleDataSet
```

### Property / 변수 / 메서드 이름

```lua
Property:
    -- 동기화되는(서버→클라) 게임 상태: PascalCase
    [Sync]
    number MaxHealth = 100
    [Sync]
    number CurrentHealth = 100

    -- 동기화하지 않는 로컬/내부 상태: camelCase
    [None]
    boolean isInvincible = false

    -- 프로퍼티 에디터로 주입하는 데이터/참조: PascalCase
    [None]
    number MoveSpeed = 5
    [None]
    Entity HealthBarEntity = nil

Method:
    -- 메서드: PascalCase, 타입 우선 선언
    [server]
    void TakeDamage(number damage)
    {
        -- ...
    }

    -- 이벤트 핸들러: Handle + 이벤트이름
    HandlePlayerDamagedEvent(PlayerDamagedEvent event) { }
```

규칙 요약입니다.
- **메서드**: `PascalCase` (`TakeDamage`, `ResetHealth`)
- **동기화 게임 상태 Property**: `PascalCase` (`CurrentHealth`, `MaxHealth`) — 외부에서 읽히는 공개 상태이기 때문입니다.
- **로컬/내부 플래그 Property**: `camelCase` (`isInvincible`, `isSunrise`)
- **지역 변수 / 매개변수**: `camelCase` (`targetEntity`, `damage`)
- **커스텀 이벤트 타입**: `PascalCase` + `Event` 접미사 (`PlayerDamagedEvent`)
- **이벤트 핸들러 메서드**: `Handle` + 이벤트명 (`HandlePlayerDamagedEvent`)
- **프레임 누적용 임시 값**: `self._T.xxx` (동기화되지 않는 임시 테이블) — `self._T.elapsedTime`

## 🧩 1. 모듈식 설계 패턴

### ✅ 좋은 예시: 독립적인 Health 모듈

데이터는 `[None]` 프로퍼티로 분리하고, 외부와는 **이벤트로만** 통신합니다. 이 컴포넌트는 어떤 엔티티에 붙여도 그대로 동작합니다.

```lua
-- HealthComponent  (재사용 가능한 순수 로직 모듈, 서버 권위)

Property:
    -- 데이터 분리: 프로퍼티 에디터에서 엔티티별로 편집합니다 (코드 수정 불필요)
    [None]
    number MaxHealth = 100

    -- 동기화되는 게임 상태 (서버 → 클라이언트)
    [Sync]
    number CurrentHealth = 100

Method:
    [server only]
    void OnInitialize()
    {
        self.CurrentHealth = self.MaxHealth
    }

    -- 공개 API: 명확하고 단순합니다. 외부 모듈을 전혀 모릅니다.
    [server]
    void TakeDamage(number damage)
    {
        if self.CurrentHealth <= 0 then return end

        self.CurrentHealth = math.max(0, self.CurrentHealth - damage)

        -- 무엇을 할지는 모릅니다. "맞았다"는 사실만 이벤트로 알립니다.
        local event = PlayerDamagedEvent()
        event.CurrentHealth = self.CurrentHealth
        event.MaxHealth = self.MaxHealth
        self.Entity:SendEvent(event)

        if self.CurrentHealth <= 0 then
            self.Entity:SendEvent(EnemyDiedEvent())
        end
    }

    [server]
    void Heal(number amount)
    {
        if self.CurrentHealth <= 0 then return end
        self.CurrentHealth = math.min(self.MaxHealth, self.CurrentHealth + amount)
    }

    [server]
    void ResetHealth()
    {
        self.CurrentHealth = self.MaxHealth
    }
```

> 포인트: `HealthComponent`는 UI도, 애니메이션도, 다른 컴포넌트도 모릅니다. `CurrentHealth`(`[Sync]`)와 이벤트만 외부로 노출합니다. → 다른 월드에 복사해 붙여도 동작합니다.

### ❌ 나쁜 예시: 결합도가 높은 코드

```lua
-- BadHealthComponent (이렇게 하지 마세요!)

Method:
    [server]
    void TakeDamage(number damage)
    {
        self.hp = self.hp - damage

        -- ❌ GUID 하드코딩으로 외부 엔티티를 끌어옴 → 다른 월드에서 재사용 불가
        local uiEntity = _EntityService:GetEntity("1f6dcd10-6306-4a47-bd41-f0a897685b73")
        uiEntity.TextComponent.Text = tostring(self.hp)   -- ❌ 뷰 로직이 로직에 섞임

        -- ❌ 다른 모듈을 직접 호출 → 그 모듈이 없으면 깨짐
        self.Entity.PlayerAnimationComponent:PlayHurtAnimation()
    }
```

문제점은 다음과 같습니다.
1. GUID 하드코딩 → 이 엔티티가 없는 다른 월드에서 즉시 깨집니다.
2. 뷰(UI 텍스트) 갱신이 로직에 섞임 → 시뮬레이션/뷰 분리 위반입니다.
3. 다른 컴포넌트를 직접 호출 → 강결합입니다. 그 컴포넌트가 없으면 에러가 납니다.
4. 서버에서 UI(클라이언트 개념)를 직접 건드림 → Local 개념 위반입니다(모든 클라이언트에 영향을 줍니다).

## 🔗 2. 모듈 연결 패턴

**이벤트를 매우 적극적으로 활용합니다.** 모듈 간 통신은 직접 호출이 아니라 **Entity Event System**으로 합니다.

### 2-1. 커스텀 이벤트 타입 정의

`Workspace - MyDesk > Create EventType`으로 이벤트를 만들고 전달할 데이터를 Property로 선언합니다.

```lua
-- PlayerDamagedEvent (EventType)
Property:
    number CurrentHealth = 0
    number MaxHealth = 0
```

### 2-2. 이벤트 발송 (Sender)

엔티티를 중계소로 사용합니다. `self.Entity:SendEvent(event)`를 호출합니다. (맵/월드 엔티티를 공용 중계소로 쓰는 경우가 많습니다.)

```lua
local event = PlayerDamagedEvent()
event.CurrentHealth = self.CurrentHealth
event.MaxHealth = self.MaxHealth
self.Entity:SendEvent(event)
```

### 2-3. Connector Component (접착제 30%)

여러 모듈을 조율하는 **전용 스크립트**입니다. 모듈 참조는 `[None]` 프로퍼티로 주입받고, 이벤트를 받아 뷰/사운드를 조율합니다. 로직은 절대 여기 두지 않습니다.

```lua
-- PlayerHealthConnector (접착제: 모듈을 연결하기만 합니다)

Property:
    -- 프로퍼티 에디터에서 주입받는 모듈/뷰 참조 (하드코딩 금지)
    [None]
    Entity HealthBarEntity = nil
    [None]
    string HurtSoundRUID = ""

Event Handler:
    -- HealthComponent가 쏜 이벤트를 받아 뷰 여러 개를 조율합니다
    [entity: self]
    HandlePlayerDamagedEvent(PlayerDamagedEvent event)
    {
        -- 체력바 갱신
        self.HealthBarEntity.HealthBarComponent:UpdateBar(event.CurrentHealth, event.MaxHealth)
        -- 피격 연출/사운드
        self.Entity.PlayerAnimationComponent:PlayHurtAnimation()
        _SoundService:PlaySound(self.HurtSoundRUID, 1)
    }

    [entity: self]
    HandleEnemyDiedEvent(EnemyDiedEvent event)
    {
        self.Entity.PlayerAnimationComponent:PlayDeathAnimation()
        -- 게임 오버 처리 등...
    }
```

> 이벤트 구독은 Event Handler를 등록하는 것으로 끝납니다. 핸들러 상단의 중계 엔티티(`[entity: self]`, `[entity: map01]`, `[service: InputService]`)를 정확히 지정하는 것이 핵심입니다 — 어느 엔티티를 중계소로 쓰느냐에 따라 누구의 이벤트를 받는지가 결정됩니다.

### 2-4. 연결 시 안전장치 — `isvalid()`

참조한 엔티티/컴포넌트는 **반드시 유효성 검사**를 합니다. MSW의 삭제는 즉시 일어나지 않으므로(삭제 요청 → 대기 → 처리 → 완료) `~= nil`이 아니라 `isvalid()`를 씁니다.

```lua
local target = _EntityService:GetEntity(self.TargetId)
if isvalid(target) == false then return end
-- 안전하게 사용
```

## 📊 3. 코드와 데이터 분리

### 3-1. 밸런싱 수치 → `[None]` 데이터 프로퍼티

코드를 고치지 않고 프로퍼티 에디터에서 엔티티별로 조정합니다.

```lua
Property:
    [None]
    number MaxHealth = 100
    [None]
    number HealthRegenRate = 5
    [None]
    boolean CanRegenerate = true
    [None]
    number BaseDefense = 0
```

### 3-2. 표시 텍스트 / 번역 → LocaleDataSet

**UI나 NPC 이름 등 유저에게 보이는 모든 텍스트는 코드에 박지 않습니다.** LocaleDataSet에 키로 정리하고 `_LocalizationService:GetText(key)`로 불러옵니다.

```lua
-- ❌ 나쁜 예
self.Entity.TextComponent.Text = "E를 눌러 상호작용"

-- ✅ 좋은 예 (LocaleDataSet에 UI_INTERACT_HINT 키 등록)
[client only]
void OnBeginPlay()
{
    self.Entity.TextComponent.Text = _LocalizationService:GetText("UI_INTERACT_HINT")
}
```

LocaleDataSet 예시입니다.

| Key | ko | en |
|---|---|---|
| UI_INTERACT_HINT | E를 눌러 상호작용 | Press E to interact |
| UI_TEXT_NPC_Stan | 스탄 | Stan |

> 리소스(이미지 등)도 RUID를 LocaleDataSet에 넣어 언어별로 다르게 줄 수 있습니다. 서버는 **번역 결과가 아니라 로컬라이징 키**를 보내고, 번역은 클라이언트에서 수행합니다(글로벌 원빌드 원칙).

### 3-3. 영속 데이터 → DataStorage

세이브/누적 데이터는 DataStorage를 사용합니다. **`GetAndWait`/`SetAndWait`의 남발을 금지**합니다(직렬 대기로 흐름이 막히고 느려집니다). 가능하면 `GetAsync`/`SetAsync` + 콜백을 씁니다. (아래 4번과 연결됩니다)

## ⚙️ 4. 실행 순서 제어

### 4-1. 라이프사이클 순서

```
OnInitialize → OnBeginPlay → OnUpdate(매 프레임) → OnMapEnter/OnMapLeave(맵 이동마다) → OnEndPlay → OnDestroy
```

| 함수 | 시점 | 용도 / 주의 |
|---|---|---|
| `OnInitialize` | 컴포넌트 생성 직후 1회 | **값 초기화 전용입니다.** 다른 컴포넌트 참조 시 nil 가능 → 참조 금지 |
| `OnBeginPlay` | 모든 엔티티/컴포넌트 생성 후 1회 | 다른 컴포넌트 참조 보장. 단 **컴포넌트 간 호출 순서는 미보장** |
| `OnUpdate(number delta)` | 매 프레임 | 최적화 민감 구역. 동기화 프로퍼티/실행제어/`wait()` 사용 주의 |
| `OnMapEnter/Leave(Entity map)` | 맵 진입/퇴장마다 | 맵별 초기화/정리 |
| `OnEndPlay` / `OnDestroy` | 제거 시점 1회 | **타이머·구독 정리(`ClearTimer`)** |
| `OnSyncProperty(string name, any value)` | `[Sync]` 값이 클라에 동기화될 때 | 클라이언트 뷰 갱신 (5번 참고) |

### 4-2. 컴포넌트 간 순서가 필요하면 `OnInitialize`에 값을 둡니다

`OnBeginPlay`끼리는 순서가 보장되지 않습니다. A의 값을 C가 `OnBeginPlay`에서 읽어야 한다면 다음과 같이 합니다.

```lua
-- ComponentA: 값 할당은 OnInitialize에서
Method:
    [server only]
    void OnInitialize() { self.B = "Hello" }

-- ComponentC: 참조는 OnBeginPlay에서 (이때 A의 OnInitialize는 이미 끝나 있음)
Method:
    [server only]
    void OnBeginPlay()
    {
        local a = self.Entity.ComponentA
        log(a.B)  -- "Hello" 보장
    }
```

### 4-3. 타이머/지연은 `_TimerService`로 — `wait()`·코루틴 오용 금지

> **규칙: 비동기/코루틴을 이용한 임의의 타이머 제어나 로직 대체를 엄격히 금지합니다. 지연·반복 실행은 `_TimerService`로 처리합니다.**

`wait()`는 **직렬(대기)**, `_TimerService`는 **병렬(예약)**입니다. `wait()` 남발은 흐름 파악을 어렵게 하고, 대기 이후 엔티티 유효성을 보장하지 못하며, 특히 `OnUpdate`에서 쓰면 Update 객체가 쌓여 월드를 느리게 만듭니다.

```lua
-- ❌ 나쁜 예: wait로 지연 흉내 (대기 이후 유효성 보장 못 함, 흐름 막힘)
[server]
void DoLater()
{
    wait(3)
    self.Entity.TestComponent:ShowMessage()  -- 3초 뒤 엔티티가 살아있다는 보장이 없음
}

-- ✅ 좋은 예: TimerService로 예약 (병렬, 비차단)
[server]
void DoLater()
{
    local function callback()
        if isvalid(self.Entity) == false then return end
        self.Entity.TestComponent:ShowMessage()
    end
    _TimerService:SetTimerOnce(callback, 3)         -- 3초 후 1회
}
```

`_TimerService` API입니다.

```lua
-- 범용: SetTimer(owner, callback, interval, isRepeat, startDelay?)
local timerId = _TimerService:SetTimer(self, callback, 3, true, 2)  -- 2초 후 첫 실행, 이후 3초마다
if timerId == 0 then return end                                     -- 실패 시 0

-- 단축형
_TimerService:SetTimerOnce(callback, 3)            -- 3초 후 1회
_TimerService:SetTimerRepeat(callback, 3, 2)       -- 2초 후 시작, 3초마다 반복

-- 반복 타이머는 반드시 정리
_TimerService:ClearTimer(timerId)
```

```lua
-- 반복 타이머는 OnEndPlay에서 정리합니다
Method:
    [server only]
    void OnBeginPlay()  { self._T.timerId = _TimerService:SetTimerRepeat(function() self:Tick() end, 1) }
    [server only]
    void OnEndPlay()    { _TimerService:ClearTimer(self._T.timerId) }
```

> 컴포넌트 메서드를 콜백으로 넘길 때는 `self:Func()`가 아니라 클로저(`function() self:Func() end`)로 감싸거나 `self.Func` 형태로 전달합니다.

### 4-4. `OnUpdate` 최적화 3원칙

`OnUpdate`는 매 프레임 호출되므로 가장 흔한 성능 문제의 원천입니다.

1. **동기화 프로퍼티를 매 프레임 변경 금지합니다.** `self.SyncedValue = self.SyncedValue + delta`는 매 프레임 N×N 네트워크 비용입니다. 누적은 비동기 로컬 값(`self._T.elapsedTime`)으로 하거나, 시작 시각만 한 번 동기화하고 클라에서 경과를 계산합니다.
   ```lua
   -- 이벤트 시작 시(Server): 한 번만 동기화
   self.eventActiveTime = _UtilLogic.ServerElapsedSeconds
   -- 경과 시간이 필요할 때(Client): 계산으로 구함
   local period = _UtilLogic.ServerElapsedSeconds - self.eventActiveTime
   ```
2. **`OnUpdate` 안에서 다른 실행 공간 호출(서버↔클라)을 금지합니다.** 매 프레임 네트워크 비용이 폭증합니다.
3. **`OnUpdate` 안에서 `wait()`를 금지합니다.** Update 객체가 쌓여 메모리/속도가 악화됩니다.

### 4-5. 특정 클라이언트에만 응답

`[client]` 함수의 마지막 매개변수에 `userId`를 넘기면 그 유저에게만 전송됩니다(자동으로 붙으므로 호출부에서 생략 가능). 모든 클라이언트에 브로드캐스트하지 않습니다.

```lua
[server]
void RequestBuyItem(integer itemId)
{
    -- ... 서버 검증/처리 ...
    self:ResponseBuyItem(itemId, self.Entity.PlayerComponent.UserId)  -- 요청자에게만
}

[client]
void ResponseBuyItem(integer itemId)  -- 실제로는 (..., string targetUserId = nil), userId 자동 부착
{
    log("Success")
}
```

## 🎭 5. 시뮬레이션과 뷰 분리

MSW에서 이 분리는 곧 **서버(시뮬레이션) ↔ 클라이언트(뷰)** 분리이며, `[Sync]` 프로퍼티와 `OnSyncProperty`가 둘을 잇습니다.

### 5-1. Simulation = 서버 권위 로직

순수 게임 상태와 로직만 둡니다. 권위 값은 `[Sync]`로 선언해 서버가 단독으로 변경합니다.

```lua
-- CombatComponent (시뮬레이션 / 서버 권위)

Property:
    [Sync]
    number Health = 100
    [None]
    number MoveSpeed = 5    -- 데이터

Method:
    [server]
    void TakeDamage(number damage)
    {
        self.Health = math.max(0, self.Health - damage)   -- 값만 바꿉니다. 뷰는 모릅니다.
    }

    [server only]
    void OnUpdate(number delta)
    {
        -- 위치/상태 등 권위 로직만 둡니다. 이펙트/사운드 호출은 없습니다.
    }
```

### 5-2. View = 클라이언트 표현 (`OnSyncProperty`)

클라이언트는 동기화된 값을 **읽어서** 시각/청각으로만 표현합니다. 권위 값을 직접 바꾸지 않습니다.

```lua
-- CombatViewComponent (뷰 / 클라이언트 표현)

Method:
    -- 서버의 [Sync] 값이 클라로 동기화되는 시점에 호출됩니다
    [client only]
    void OnSyncProperty(string name, any value)
    {
        if name == "Health" then
            self.Entity.HealthBarComponent:UpdateBar(value)   -- UI 갱신
            if value <= 0 then
                self.Entity.AnimationComponent:PlayDeath()    -- 이펙트/연출
            end
        end
    }
```

### 5-3. Local(내 캐릭터/입력/UI) 처리

내 캐릭터·입력·UI Entity는 **내 클라이언트에만 존재하는 로컬 개념**입니다. 모든 클라이언트에서 실행되는 핸들러에서는 **내 엔티티인지 반드시 확인**합니다. 안 그러면 남이 키를 눌러도 내 캐릭터가 반응합니다.

```lua
Event Handler:
    [service: InputService]
    HandleKeyDownEvent(KeyDownEvent event)
    {
        -- 내 플레이어가 아니면 무시 (멀티플레이 필수 가드)
        if self.Entity ~= _UserService.LocalPlayer then return end

        if event.key == KeyboardKey.S then
            self:UseUltimateSkill()
        end
    }
```

추가로 주의할 점입니다.
- **UI Entity는 서버에 존재하지 않습니다(Localized Entity).** UI에서 서버 실행제어 함수를 직접 호출할 수 없으므로, World Entity를 거쳐 통신합니다.
- 서버 `OnBeginPlay`에서 클라이언트로 정보를 보내면 안 됩니다(그 시점에 클라이언트가 미접속 상태입니다). 클라이언트가 자신의 `OnBeginPlay`에서 서버에 "나 준비됐다"를 알리고 받는 구조로 짭니다.
